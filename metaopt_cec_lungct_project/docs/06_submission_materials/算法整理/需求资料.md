## 一、必须有的 6 类证据

### 1. 统计显著性检验

只看均值提升还不够，必须回答：

- 这个提升是不是偶然波动？
- 不同函数上的优势是不是“显著”而不是“看起来大”？

常见做法就是：

- **Wilcoxon signed-rank test（威尔科克森符号秩检验）**
- **Friedman rank test（弗里德曼排序检验）**

你手头的 BBO 论文本身就是这么做的：在不同维度 CEC 上，不只看结果均值和标准差，还做了 Wilcoxon 和 Friedman 来验证统计显著性。
SBO 论文也明确把这两类检验当作“可靠性验证”的核心部分。

**对你当前版本的意义：**
如果 ROUTE_A_BUDGET_ADAPTIVE_BBO 只是“平均值略好”，但显著性不成立，那论文里不能写成“明显优于”。

------

### 2. 稳定性 / 鲁棒性分析

这个其实是你现在特别要重视的。

不是只看 **mean（均值）**，还要看：

- **std（标准差）**
- 胜/平/负数量
- 箱线图（boxplot）
- 不同维度、不同函数类别下是否稳定

BBO 论文就强调了 **mean + std + boxplot + convergence** 一起看，来说明“不是偶然跑得好一次，而是整体更稳”。
你自己的科研工作流里也把“统计/可视化”明确列为主线任务。

**对你这里最关键的点：**

- 你要特别拆开看 **F10-F12 这类难函数** 和其他函数
- 看是不是“有些函数大幅变好，但另一些函数明显变差”
- 看是不是“均值提升了，但 std 爆了”，那就说明方法不稳

------

### 3. 消融实验（Ablation）

这个是你现在最不能缺的。

因为你这个版本叫 **ROUTE_A_BUDGET_ADAPTIVE_BBO**，那评审一定会问：

- 到底是 **Budget adaptive（预算自适应）** 起作用？
- 还是其他伴随改动起作用？
- 它单独值不值得保留？

所以你至少要有：

- `BBO_BASE`
- `BBO + budget_adaptive`
- 如果后面继续加 archive / replay / success-history，也要继续拆开：
  - `BASE + budget_adaptive`
  - `BASE + budget_adaptive + archive`
  - `BASE + budget_adaptive + archive + replay`
  - …

你仓库规则里也明确要求：**baseline、improvement modules、full improved version 必须清晰区分，改进模块要能单独开关，便于消融。**

**一句话理解：**
对比实验回答“你比别人强不强”，
消融实验回答“你到底是靠什么变强的”。

------

### 4. 参数敏感性分析

你这版本质上是“策略型改进”，所以不能只证明“某个参数正好凑巧有效”，还要证明：

- 参数改一点还行不行
- 不同预算下是否依然有效
- 不同 FEs（Function Evaluations，函数评估次数）下趋势是否一致

你自己的项目主线就把 **参数敏感性实验**单独列出来了。

你这里最该做的敏感性，不是那种很泛的全参数扫，而是：

- **预算切换阈值**
- **阶段分配比例**
- **早期/中期/后期 budget 的占比**
- **低预算（3e4）与高预算（3e5）下是否一致**

**为什么特别重要：**
因为 `budget_adaptive` 的核心卖点不是“某个固定配置赢”，而是“预算分配策略本身更合理”。

------

### 5. 收敛行为 + 计算效率分析

改进算法不能只说“最后结果更好”，还要回答：

- 是不是 **更快收敛**
- 还是只是 **更晚但更低**
- 前期是否更稳
- 后期是否更能继续下降
- 有没有代价，比如额外耗时、额外内存、额外复杂度

BBO 论文和 SBO 论文都很重视这块：
BBO 会看 **convergence curves（收敛曲线）**、boxplots、稳定性；
SBO 还专门比较了 **runtime（运行时间）** 和高维下的可扩展性。

你这里建议至少补：

- 平均收敛曲线
- 若干代表函数的单函数收敛图
- 平均耗时 / 单次运行耗时
- 若有额外算子开销，给出“精度收益 vs 计算代价”说明

------

### 6. 公平性与可复现性说明

这一类最容易被忽视，但论文里很要命。

你仓库说明明确强调：

- 不要悄悄改实验协议
- 要控制随机种子
- 要统一实验入口
- 要保留原始结果
- baseline / improved / ablation / final 要可区分
- 不能破坏算法间可比性。

所以你要补的不是“再跑更多”，而是把这些写清楚：

- population size 是否一致
- FEs 是否一致
- run 数是否一致
- seed 策略是否一致
- 边界处理、停止准则是否一致
- 是否只改了算法内部，不改 benchmark protocol

**一句话：**
没有公平协议，结果再好也容易被质疑“比的不公平”。

## 二、建议补的 3 类增强证据

### 7. 分函数类别的归因分析

这个对你特别重要。

别只给总表，要按类别拆：

- unimodal（单峰）
- multimodal（多峰）
- hybrid（混合）
- composition（组合）

因为你最近的问题本来就集中在某些复杂函数上。
SBO 论文也明确指出，不同函数类别代表不同能力：单峰偏 exploitation（开发能力），多峰偏 escape local optima（跳出局部最优），hybrid/composition 更能看综合平衡能力。

所以你应该回答：

- `budget_adaptive` 是不是主要提升了 **hybrid / composition**
- 有没有牺牲 unimodal
- 有没有在低预算和高预算下结论相反

这一步非常适合写论文里的“结果分析”小节。

------

### 8. 失败案例 / 负面结果分析

这个特别像“研究味”。

如果某些函数上 `ROUTE_A_BUDGET_ADAPTIVE_BBO` 没提升，甚至变差，要单独说：

- 失败集中在哪类函数
- 是前期 exploration（探索）不足，还是后期 exploitation（开发）不足
- 是被局部最优卡住，还是预算分配过早收缩
- 是均值差，还是 std 爆炸

你前面几轮其实已经在做这件事了，只是现在可以把它正式化成：

- “优势函数簇”
- “无效函数簇”
- “退化函数簇”

这会让后续 archive / replay / success-history 的设计更有根据。

### 9. 应用验证 / 泛化验证

你仓库主线明确写了：**肺 CT 应用验证是应用案例，不是核心 benchmark 主线，但它是后续必须有的一环。**

所以对你当前阶段来说：

- **CEC 先证明算法本体有效**

这个和通用科研流程也一致：分析与验证阶段除了量化、统计、消融，还要关注鲁棒性与可推广性。

你现在如果问“普遍还需要什么”，答案里**应用验证算加分项/后验项**，但不是你当前最优先补的第一个。
