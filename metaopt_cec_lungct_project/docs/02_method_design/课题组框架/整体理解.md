### 一、这个框架的整体组成

从代码结构上看，它大致分成 5 层。

| 层   | 目录/文件           | 作用             | 对你项目的价值 |
| ---- | ------------------- | ---------------- | -------------- |
| 1    | `mealpy/`           | 算法库           | 很高           |
| 2    | `opfunu/`           | CEC 基准函数库   | 很高           |

#### 1）`mealpy/`：算法层

这是整个框架的核心算法层，里面按类别组织了大量优化算法实现。
我静态统计到它大约有 **130 个算法实现文件**，主要分为：

- `bio_based/`（生物启发）
- `evolutionary_based/`（进化类）
- `human_based/`（人类行为启发）
- `math_based/`（数学启发）
- `physics_based/`（物理启发）
- `swarm_based/`（群体智能）
- `mdm_group/`（项目自定义/新增的一组算法）
- 以及 `music_based/`、`system_based/`

其中 `mdm_group/` 里放的是这个课题组重点加入的算法，例如：

- `SBO.py`
- `RIME.py`
- `PLO.py`
- `RUN.py`
- `INFO.py`
- `HGS.py`
- `SMA.py`

**这层最值得你学的是“统一算法接口”设计。**

------

#### 2）`mealpy/optimizer.py`：算法抽象基类（Base Class，基类）

这个文件是全框架最关键的“骨架”。

它把所有算法统一成一个范式：

- `initialization()`：初始化种群
- `evolve(epoch)`：每一代如何更新
- `solve(problem, ...)`：统一求解入口
- `history`：统一记录收敛过程
- `problem`：统一问题定义（目标函数、边界、最小化/最大化）

这意味着：

> 不同算法虽然公式不同，但都能用同一套调用方式跑实验。

这对你的课题特别重要，因为你现在其实已经进入了**“多版本改进算法并行比较”**阶段。
如果没有统一接口，后面会出现两个问题：

- 改一个算法就要改一套实验脚本
- 消融实验（ablation，删模块验证贡献）会越来越乱

**对你项目的直接借鉴：**
你后面无论是做 HHO 改进版，还是 Beaver Behavior Optimizer（河狸行为优化）改进版，都应该统一成类似：

- `initialize_population()`
- `evaluate_population()`
- `evolve_one_iteration()`
- `solve()`

这样后面做：

- baseline（原始算法）
- improved（完整改进版）
- ablation（去模块版）
- sensitivity（参数敏感性）

就会非常顺。

------

#### 3）`opfunu/`：基准函数层

这个目录本质上是**测试题库**。
它封装了很多经典优化测试函数，尤其是：

- `CEC2005`
- `CEC2017`
- `CEC2019`
- `CEC2022`

`run_exp.py` 里通过动态导入（dynamic import，运行时导入）的方法自动加载指定 benchmark（基准套件），再自动识别函数类名，比如：

- `F12017`
- `F22017`
- ...
- `F302017`

这部分的优点是：

- **不用你手写每个函数**
- **能统一切换 benchmark 年份**
- **利于保证测试标准一致**

**对你项目的借鉴：**
你现在做改进算法，最需要的就是**严格统一的 CEC 测试入口**。
这套思路非常适合你把当前 MATLAB（矩阵实验）流程，迁移成一个 Python（Python 编程语言）版标准框架：

- 先选 benchmark 套件
- 再选函数集合
- 再选维度
- 再选算法列表
- 最后统一输出结果

------

#### 4）`run_exp.py`：主实验入口

这是这个框架真正“把算法跑起来”的主控脚本。

它做的事情可以概括成：

1. 选 benchmark（比如 `CEC2017`）
2. 设定维度、种群数、迭代数、运行次数
3. 选算法列表（默认是 `PLO`、`RIME`、`SBO`）
4. 动态加载所有测试函数
5. 对每个函数做多次独立运行
6. 记录最优值和收敛曲线
7. 画收敛图
8. 导出 Excel（Excel 电子表格）和 boxplot（箱线图）

这个文件体现的是**“研究实验主控器”**思想，而不是普通业务脚本思想。

------

#### 5）`utils/utils.py`：结果分析层

这个文件很有研究味道。它不只是简单保存结果，而是做了完整的论文型输出：

- `best_fitness_all.npy`：原始数值
- `raw_data`：每次运行的原始记录
- `cmp_result`：均值、标准差、排序
- `pValue`：Wilcoxon（威尔科克森符号秩检验）
- `rank_pValue`
- `pValue_sign`
- 每个函数单独 sheet（表单）
- 收敛曲线图
- 箱线图

**这部分对你的借鉴价值非常高。**
因为你现在最大的问题不是“算法能不能跑”，而是：

> 跑出来以后，怎么形成可写论文、可复核、可比较的证据链。

这个工具层正好在解决这个问题。

------

#### 6）`application/fs.py`：真实应用示例层

这个文件不是做 CT 分割的，而是做**特征选择**（Feature Selection）的。

它的思路是：

- 读取数据集
- 用 KFold（K 折交叉验证）切分
- 用优化算法搜索“哪些特征应该保留”
- 用分类器（如 KNN、SVM、RF）评估选择结果
- 目标函数 = 分类误差 + 特征数量惩罚
- 保存每折结果、收敛曲线、最佳解

这个文件的重要价值不在于“任务本身”，而在于它展示了：

> **如何把一个真实问题，包装成优化器能处理的 objective function（目标函数）**

这对你做肺 CT 项目很有启发。

### 二、这个框架真正的执行逻辑

如果用一句话概括，它的工作方式是：

**算法层（mealpy） + 问题层（opfunu / 真实任务） + 实验层（run_exp） + 统计层（utils）**

#### 连续优化实验流

```text

→ 自动加载函数
→ 选算法
→ solve(problem)
→ 记录 fitness（适应度）/ convergence（收敛曲线）
→ 统计检验
→ 导出图表和 Excel
```

#### 真实应用实验流

```text
读数据
→ 定义 objective function
→ 用优化器搜索参数/特征子集
→ 用分类器或任务指标评估
→ 保存每折结果
→ 输出汇总统计
```

### 五、这个框架目前存在的几个问题（对你尤其重要）

### 1. 可复现性（Reproducibility，可重复性）还不够严

虽然框架目标是可复现，但当前默认脚本里，我看到两个明显问题：

#### 问题 A：没有明确固定 seed（随机种子）

`solve(problem)` 没有显式传 `seed`。
这意味着：

- 每次跑的随机初始化可能不同
- 多次 formal（正式实验）结果难完全复现

对于你的论文，这是硬伤。

#### 问题 B：默认 `RUNS = 4`

如果拿去写正式 benchmark（基准实验）论文，**4 次独立运行通常偏少**。
你后面正式实验最好用：

- 30 次
- 或至少 20 次

------

### 2. 预算定义可能不够严谨

`run_exp.py` 里：

```python
MAX_EPOCHS = 1000  # Max function evaluations
```

但从代码看，它实际上是传给：

```python
model = algo_class(epoch=MAX_EPOCHS, pop_size=SEARCH_AGENTS_NO)
```

也就是它在算法内部主要代表的是：

**迭代代数（epoch）**

不是严格意义上的：

**函数评价次数（FEs, Function Evaluations）**

这对 CEC 公平性非常关键。

因为很多论文比较时要求统一的是：

- 最大函数评价次数
- 而不是仅仅代数

如果不统一 FE（Function Evaluations，函数评价次数），不同算法之间的比较可能不完全公平。

**这一点对你的项目特别重要**，因为你现在正处在“算法改进要拿 benchmark 说话”的阶段。

------

### 3. benchmark 和 application 耦合还不够彻底分离

如果你以后项目扩展到：

- CEC
- lung CT
- 多阈值分割
- 参数敏感性
- 消融实验

建议你做得比它更进一步，把配置统一起来。
