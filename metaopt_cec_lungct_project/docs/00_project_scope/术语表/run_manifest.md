### 直接理解

**`run_manifest` 不是算法术语，而是实验管理里的“运行清单 / 运行登记表”。**

从你这个项目的约束文档看，它和 `raw_runs`、`summary`、`logs` 被并列要求“可落盘”，也就是一次实验跑完后，必须作为结果产物保存下来，用来支撑**复现性**。

### 它在这个项目里大概是干什么的

结合文档上下文，`run_manifest` 的作用更像是：

- **记录这次实验到底跑了什么**
- **把配置、结果文件、日志文件串起来**
- **让后面能追溯某个结果是怎么来的**

因为文档明确要求结构类改动的验收里，要同时检查 `raw_runs`、`summary`、`run_manifest`、`logs` 是否都能落盘，这说明 `run_manifest` 不是可有可无的附属文件，而是**复现实验链路的一部分**。

### 可以把它理解成什么

你可以把它理解成：

> **一次实验批次的“总目录说明书”或“索引文件”**

它通常会回答这几个问题：

- 这次实验的名称是什么
- 跑的是哪个 suite（基准套件）
- 跑了哪些函数、维度、算法
- 一共跑了多少次 run
- 随机种子是什么
- 生成了哪些文件
- `summary.csv`、`raw_runs/`、`logs/` 分别在哪里
- 哪个结果对应哪个配置

### 它和其他结果文件的区别

简单区分一下：

- **`raw_runs/`**：放每次单独运行的原始结果
- **`summary`**：放汇总统计，比如 mean（均值）、std（标准差）、best（最优）这些
- **`logs`**：放运行日志、报错、时间信息
- **`run_manifest`**：放“这批实验整体的登记信息和文件映射关系”

所以它更像一个**总索引**，不是原始结果本身。
