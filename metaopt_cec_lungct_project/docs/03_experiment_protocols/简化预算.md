### 先给结论

你这套主线配置：

```matlab
dim = 10
pop_size = 30
maxFEs = 300000
```

**确实偏慢了**。
如果目标是：

- **先做母算法改进验证**
- **保证结果还有参考价值**
- **不要一上来把时间全烧光**

那我建议把主线配置改成下面这套：

```matlab
cfg.dim = 10;
cfg.pop_size = 30;
cfg.maxFEs = 100000;
cfg.runs = 10;
```

这套更适合你当前阶段，原因是：

- `D=10`（维度）本来就不高
- 对 CEC（Congress on Evolutionary Computation，进化计算大会）类连续优化测试，常见公平预算是 **10000 × D**
- 所以 `D=10` 时，**100000 FEs（函数评价次数）** 是一个很标准、也很能被接受的级别
- 你现在的 `300000` 相当于 **30000 × D**，更像“加强版预算”，适合后期精修，不适合作为当前主线默认值

### 为什么你这套会慢

你的当前算法集是：

```matlab
{'BBO', 'SBO', 'HGS', 'SMA', 'HHO', 'RUN', 'INFO', 'MGO', 'PLO', 'PO'}
```

这已经是 **10 个算法** 了。
再乘上：

- `2` 个测试集（CEC2017、CEC2022）
- 若干函数
- 多次独立运行（runs，重复次数）
- `300000` 的大预算

总开销会非常夸张。

可以粗算一下：

假设：

- 2 个 suite（测试套件）
- 每套先跑 10 个函数
- 10 个算法
- 每个算法 10 次独立运行
- 每次 300000 FEs

总函数调用量约为：

```text
2 × 10 × 10 × 10 × 300000 = 600,000,000
```

就是 **6 亿次目标函数调用**。

即使单次函数很快，整体也会非常慢。
如果你还是 MATLAB（矩阵实验环境）+ 多算法 wrapper（包装器）框架，这个量级足够让调试期很痛苦。

### 我建议你直接把默认配置改成什么

如果你现在问的是：

> “我要把 `fill_default_config` 里的主线默认值改成什么最合适？”

那我建议改成：

```matlab
function cfg = fill_default_config(cfg)
    if ~isfield(cfg, 'suites')
        cfg.suites = {'cec2017', 'cec2022'};
    end
    if ~isfield(cfg, 'algorithms')
        cfg.algorithms = {'BBO', 'SBO', 'HGS', 'SMA', 'HHO'};
    end
    if ~isfield(cfg, 'func_ids')
        cfg.func_ids = [];
    end
    if ~isfield(cfg, 'dim')
        cfg.dim = 10;
    end
    if ~isfield(cfg, 'pop_size')
        cfg.pop_size = 30;
    end
    if ~isfield(cfg, 'maxFEs')
        if isfield(cfg, 'max_fes')
            cfg.maxFEs = cfg.max_fes;
        else
            cfg.maxFEs = 100000;
        end
    end
    if ~isfield(cfg, 'runs')
        cfg.runs = 10;
    end
end
```

------

### 为什么不是继续保留 300000

因为 `300000` 更适合这两种场景：

#### 1. 少量函数精修验证

比如你已经筛出几类最关键函数：

```matlab
cfg.func_ids = [6 9 15 21 28];
cfg.maxFEs = 300000;
```

这种可以。
因为你不是全函数、全算法铺开跑，而是做重点验证。

#### 2. 后期证明“高预算下仍然有效”

这适合论文增强结果，比如：

- 标准预算下已经赢
- 更高预算下依旧稳定

这个时候 `300000` 是“附加说服力”，不是主线默认值。

------

### 你现在最应该避免的错误

#### 1. 一上来就全算法 + 双套件 + 全函数 + 30 runs + 300000 FEs

这是最耗时间的组合。
非常容易出现：

- 调试效率极低
- 小 bug 修一次等半天
- 结果目录爆炸
- 还没开始做改进就先被计算拖死

#### 2. 为了快把 `pop_size` 压太低

比如压到 10 或 15。
这样虽然更快，但很多群智能算法（Population-based Optimization Algorithm，基于种群的优化算法）会明显失真。

所以我建议：

- `pop_size = 30` 保持不动
- 优先削减 `maxFEs`
- 再削减 `runs`
- 最后才考虑缩算法数量

#### 3. 默认把所有算法都开着

这不是主线开发的好习惯。
应该分层：

- **开发层**：少量核心算法
- **验证层**：中等数量算法
- **论文层**：全算法
