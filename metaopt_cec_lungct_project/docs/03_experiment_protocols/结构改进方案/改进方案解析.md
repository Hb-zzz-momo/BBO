### 先说结论

这些都**不是 MATLAB（矩阵实验室）官方固定语法**，而是项目里常见的**文件命名约定**。
它们本质上是在表达：**这个文件在整个调用链里扮演什么角色**。

可以把它理解成一条流水线：

**入口文件 → 统一调度 → 分发管理 → 接口适配/包装 → 真正算法实现**

------

### 一、逐个解释

| 文件名                 | 通常意思                   | 主要职责                           | 一般是否写核心算法 |
| ---------------------- | -------------------------- | ---------------------------------- | ------------------ |
| `run_xxx_unified.m`    | 某类任务的**统一运行入口** | 统一配置、统一输出、统一流程       | 一般不写核心算法   |
| `run_xxx_entry.m`      | 某个任务的**外部入口脚本** | 直接启动、给人点运行的             | 一般不写核心算法   |
| `api_wrapper.m`        | **接口包装层**             | 把底层复杂函数包成统一调用形式     | 一般不写核心算法   |
| `adapter_layer.m`      | **适配层**                 | 做格式转换、参数转换、老新接口兼容 | 一般不写核心算法   |
| `manager_dispatcher.m` | **管理+分发器**            | 根据模式/任务类型分配到不同模块    | 一般不写核心算法   |

------

### 二、每个名字拆开讲

#### 1. `run_xxx_unified.m`

这个名字里最关键的是：

- `run`：表示“运行脚本/运行入口”
- `xxx`：表示具体任务名，比如 `cec2017`、`lungct`、`ablation`
- `unified`：表示“统一的”

所以它一般表示：

> **把某一类实验或任务，统一成一个标准入口来运行**

比如你现在项目里可能会有：

- `run_cec_unified.m`
- `run_ct_unified.m`
- `run_ablation_unified.m`

这种文件通常会做几件事：

1. 读取配置（config，配置）
2. 判断运行模式
3. 调用对应算法
4. 收集结果
5. 导出表格/图/日志

它更像是一个**总控脚本**，不是“算法主体”。

**通俗比喻：**
它像“总调度室”，负责把整次实验按统一流程跑完。

------

#### 2. `run_xxx_entry.m`

这里的关键词是 `entry`（入口）。

它一般表示：

> **给用户、命令行（command line，命令行）、批处理脚本直接调用的入口文件**

和 `unified` 的区别是：

- `entry` 更强调“**从哪里点进去开始运行**”
- `unified` 更强调“**内部流程是统一的**”

也就是说，`run_xxx_entry.m` 往往是最外层那一下：

```matlab
run_xxx_entry -> run_xxx_unified -> 更底层模块
```

它通常很薄，只做：

- 设默认参数
- 调一次统一入口
- 显示提示信息

**通俗比喻：**
它像“楼的大门”，你先进门；
而 `run_xxx_unified.m` 像“大楼里的总服务台”。

------

#### 3. `api_wrapper.m`

这里的 `API` 是 **Application Programming Interface（应用程序编程接口）**。
`wrapper` 是“包装器”。

它一般表示：

> **把底层原本不好直接调用的函数，包装成一个统一、稳定、容易被外部调用的接口**

比如底层算法原来参数很乱：

```matlab
[best, curve] = BBO_improved_v3(N, T, lb, ub, dim, fobj)
```

但上层框架可能想统一成：

```matlab
result = api_wrapper(cfg)
```

那 `api_wrapper.m` 就会负责：

- 从 `cfg`（配置结构体）里拆参数
- 调底层函数
- 把输出整理成统一结构

所以它的作用是：

**让上层不用知道底层细节。**

**通俗比喻：**
像“前台窗口”。
外面的人只管按统一表格办事，窗口后面怎么对接内部系统不用管。

------

#### 4. `adapter_layer.m`

`adapter` 是“适配器”，`layer` 是“层”。

它一般表示：

> **专门负责把“这边的格式”转成“那边的格式”**

比如常见的适配包括：

- 参数名不一致
- 输入输出结构不一致
- 旧代码接口和新框架接口不一致
- CEC（Congress on Evolutionary Computation，进化计算大会）测试接口和你自己的算法接口不一致

例如：

- 你的算法需要：`fobj, lb, ub, dim`
- 统一框架给的是：`problem_struct`

那适配层就负责把 `problem_struct` 转成你算法吃得下的格式。

所以它强调的是：

**“兼容”和“转换”**

而 `wrapper` 强调的是：

**“封装”和“对外提供统一调用方式”**

两者很像，但不完全一样。

------

#### 5. `manager_dispatcher.m`

这个名字其实是两个词拼起来：

- `manager`：管理器
- `dispatcher`：分发器/调度器

它一般表示：

> **根据条件，把任务派发到不同模块去执行**

比如它可能会判断：

- 跑的是 CEC2017 还是 CEC2022
- 跑 baseline（基线算法）还是 improved（改进算法）
- 跑 formal（正式实验）还是 smoke（冒烟测试）
- 跑 benchmark（基准测试）还是 CT（计算机断层扫描）应用

然后分发到：

- `run_cec_unified`
- `run_ct_unified`
- `run_ablation_unified`
- `run_sensitivity_unified`

所以它本质是个**路由器（router，路由器）/总分发台**。

**通俗比喻：**
像医院导诊台：
先判断你该去内科、外科还是影像科，再把你分过去。

------

### 三、它们之间最常见的关系

如果一个项目里这几个名字同时出现，常见调用链大概是：

```matlab
run_xxx_entry.m
   ↓
run_xxx_unified.m
   ↓
manager_dispatcher.m
   ↓
adapter_layer.m / api_wrapper.m
   ↓
真实算法函数（比如 BBO / HHO / CEC evaluator）
```

也可能是：

```matlab
run_xxx_entry.m
   ↓
manager_dispatcher.m
   ↓
run_xxx_unified.m
   ↓
api_wrapper.m
   ↓
core algorithm
```

所以你看到这些名字时，可以这样判断：

- **谁负责启动？** 看 `entry`
- **谁负责统一流程？** 看 `unified`
- **谁负责分流？** 看 `dispatcher`
- **谁负责兼容转换？** 看 `adapter`
- **谁负责对外统一接口？** 看 `wrapper`

------

### 四、最容易混淆的两组

#### `entry` 和 `unified` 的区别

- `entry`：更像“从哪开始”
- `unified`：更像“进去后按什么统一流程跑”

#### `wrapper` 和 `adapter` 的区别

- `wrapper`：把复杂东西包起来，给外部一个统一入口
- `adapter`：把不兼容的格式/接口转成兼容的

一句话记：

> `wrapper` 偏“封装”，`adapter` 偏“转换”。

------

### 五、放到你的科研项目里怎么理解最合适

结合你现在这种**科研实验仓库（repository，代码仓库）**场景，这类命名通常是为了服务：

- **统一实验入口**
- **可复现（reproducibility，可重复复现）**
- **公平对比**
- **批量跑实验**
- **便于后续论文写作对应方法/实验章节**

这和你项目里强调的**统一实验入口、可复现、结果导出、研究主线清晰**是同方向的。

------

### 六、一个很实用的判断方法

以后看到这种文件名，先问自己这 3 个问题：

#### 1）它是不是“真正干活的算法文件”？

如果名字里有：

- `entry`
- `unified`
- `wrapper`
- `adapter`
- `dispatcher`
- `manager`

那**大概率不是核心算法文件**，而是“组织调用流程”的。

#### 2）它是不是“中间层”？

如果它主要在做：

- 参数整理
- 模式判断
- 路径处理
- 输出整理
- 接口转换

那它就是中间层，不是算法创新点。

#### 3）它会不会造成“越叠越高”？

如果一个项目里同时有很多：

- `entry`
- `wrapper`
- `adapter`
- `manager`
- `dispatcher`

但每层都很薄，就容易形成你前面说的那种**薄层叠薄层**问题。
这时就要判断：哪些层是真有必要，哪些层只是名字高级但实际重复。

你可以直接这样背：

- `run_xxx_entry.m`：**最外层启动入口**
- `run_xxx_unified.m`：**统一实验流程入口**
- `api_wrapper.m`：**把底层函数包成统一接口**
- `adapter_layer.m`：**做输入输出/参数格式适配**
