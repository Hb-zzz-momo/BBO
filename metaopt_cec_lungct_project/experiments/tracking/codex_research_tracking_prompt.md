---

### 二、更新原则

#### 1）绝不编造
- 不要编造实验结果
- 不要编造文件路径
- 不要编造已经跑完的实验
- 不要编造“提升显著”之类结论
- 如果某个文件不存在，就写“未发现，需用户确认”

#### 2）每次更新必须包含三层信息
- **事实**：仓库里真实存在的代码、结果、文档、图表、日志
- **当前判断**：基于结果可得出的阶段性结论
- **下一步动作**：下一轮最小闭环应该做什么

#### 3）必须把“科研任务”与“文件资产”绑定
每一条进度记录都必须明确：
- 这项任务的目标是什么
- 相关代码文件有哪些
- 相关结果文件有哪些
- 相关图表/表格/统计文件有哪些
- 相关说明文档有哪些
- 当前是否可用于论文写作

#### 4）优先维护主线
围绕以下主线组织：
- baseline 算法
- 改进算法版本（如 v1/v2/v3/v4）
- 消融实验
- sensitivity（敏感性分析）
- benchmark（CEC）
- CT 应用验证
- 结果分析与论文写作支持

### 四、每次你处理任务时的工作流

每次收到新的科研任务时，按下面顺序执行：

#### 第一步：扫描并定位
先查找本次任务相关文件，包括：
- 代码
- 结果压缩包
- summary / csv / mat / json
- README / 分析文档
- 图表输出目录
- 日志

#### 第二步：判断任务归属
将本次任务归类到以下之一：
- baseline
- improved_version
- ablation
- sensitivity
- benchmark
- ct_validation
- paper_support
- repo_cleanup

#### 第三步：更新主进度表
新增或更新对应任务项，写清：
- 当前在做什么
- 已做到哪一步
- 当前阶段结论
- 卡点是什么
- 下一步是什么

#### 第四步：更新资产索引
把本次任务涉及到的代码、结果、文档都挂到同一条任务名下面。

#### 第五步：如果有结果，则输出“最小研究判断”
例如：
- 哪类函数提升明显
- 哪类函数退化明显
- 模块贡献大不大
- 结果是否支持继续该方向
- 是否适合进入 formal

#### 第六步：输出下一步建议
遵循“最小闭环优先”：
- 先修最关键问题
- 先做最能判定方向对错的实验
- 不要一次扩太多分支
